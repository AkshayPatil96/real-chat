## 🔒 Security Issues Fixed

### ❌ Before (INSECURE)
```typescript
// Raw S3 URLs were constructed and returned
const fileUrl = `https://${bucket}.s3.${region}.amazonaws.com/${fileKey}`;
return { uploadUrl, fileKey, fileUrl }; // ❌ BAD

// Database stored URLs
{
  attachment: {
    fileUrl: "https://bucket.s3.amazonaws.com/file.jpg" // ❌ BAD
  }
}
```

### ✅ After (SECURE)
```typescript
// Only keys are returned
return { uploadUrl, fileKey }; // ✅ GOOD

// Database stores keys
{
  attachment: {
    fileKey: "attachment/user123/file.jpg" // ✅ GOOD
  }
}

// URLs generated at read-time
const url = fileAccessService.getPublicUrl(fileKey); // CloudFront
```

## 🏗️ Architecture

### Service Separation

```
┌─────────────────────┐
│  UploadService      │  Write Operations
│  ─────────────────  │
│  • Upload to S3     │
│  • Delete from S3   │
│  • Presigned PUT    │
│  Returns: fileKey   │
└─────────────────────┘

┌─────────────────────┐
│ FileAccessService   │  Read Operations
│  ─────────────────  │
│  • CloudFront URLs  │
│  • Presigned GET    │
│  • Download URLs    │
│  Input: fileKey     │
└─────────────────────┘

┌─────────────────────┐
│   Controller        │  Orchestration
│  ─────────────────  │
│  Combines both      │
│  services           │
└─────────────────────┘
```

### Flow

```
Upload Flow:
Client → Controller → UploadService.generatePresignedUploadUrl()
                   → Returns: { uploadUrl, fileKey }
       → Controller → FileAccessService.getPublicUrl(fileKey)
                   → Returns: CloudFront URL
       → Response: { uploadUrl, fileKey, fileUrl }
       → Client stores fileKey in DB ✅

Display Flow:
Client → Reads fileKey from DB
       → Constructs: `${CLOUDFRONT_BASE}/${fileKey}`
       → OR calls: GET /url/:fileKey
       → Displays file ✅
```

## 🔑 Key Principles

### 1. Keys, Not URLs
```typescript
// ✅ Store this
{ fileKey: "attachment/user123/file.jpg" }

// ❌ Not this
{ fileUrl: "https://..." }
```

### 2. Upload URL ≠ Download URL
```typescript
// Upload URL (presigned PUT, 5 min expiry)
uploadUrl: "https://bucket.s3.amazonaws.com/...?signature=..."

// Download URL (CloudFront, permanent)
fileUrl: "https://cloudfront.net/attachment/user123/file.jpg"
```

### 3. Generate URLs at Read-Time
```typescript
// ❌ Wrong: Store URL at write-time
await db.save({ fileUrl: generatedUrl });

// ✅ Correct: Store key, generate URL when needed
await db.save({ fileKey });
const url = fileAccessService.getPublicUrl(fileKey);
```

## 📝 Migration (If Needed)

If existing database has `fileUrl` fields:

```typescript
// Run this script once
import { fileAccessService } from './modules/uploads/index.js';

async function migrate() {
  const messages = await Message.find({ 'attachment.fileUrl': { $exists: true } });
  
  for (const message of messages) {
    // Extract key from old URL
    const url = message.attachment.fileUrl;
    const key = url.split('.amazonaws.com/')[1] || url.split('.cloudfront.net/')[1];
    
    // Update to use key
    message.attachment.fileKey = key;
    delete message.attachment.fileUrl;
    
    await message.save();
  }
  
  console.log(`✅ Migrated ${messages.length} messages`);
}
```

## 🎓 Key Learnings

### What We Fixed
1. **Incorrect:** Storing URLs (ephemeral, tied to infrastructure)
2. **Correct:** Storing keys (permanent, infrastructure-agnostic)

### Architecture Pattern
```
Write: UploadService (S3 operations)
Read: FileAccessService (URL generation)
Controller: Combines both
```

### Security Model
```
Private S3 ← CloudFront (OAI) ← Public Internet
          ✓ Secure       ✓ Fast
```

## ✨ Summary

**Problem:** Raw S3 URLs were being stored and exposed, creating security vulnerabilities and limiting flexibility.

**Solution:** Refactored to store only file keys, generate CloudFront URLs at read-time, and separate write/read concerns.

**Result:** 
- 🔒 More secure (private S3, public CloudFront)
- 🚀 Better performance (CloudFront caching)
- 🔧 More flexible (can change CDN without DB migration)
- ✅ Backward compatible (no breaking changes)
